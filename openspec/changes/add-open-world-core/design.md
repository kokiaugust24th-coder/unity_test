# Design: オープンワールドコアシステム

## Context

- 対象環境: Unity 6 (6000.x) + URP。ターゲットは PC / コンソール(モバイルはスコープ外だが設計上排除しない)
- 参考モデル: Unreal Engine 5 の World Partition(グリッドストリーミング)、HLOD、Data Layers、Streaming Sources
- シングルプレイヤー前提だが、将来のサーバー権威型マルチプレイヤーに備えストリーミングソースを複数対応・抽象化する
- ワールド想定規模: 2km〜8km 四方、セルあたり数百オブジェクト

## Goals / Non-Goals

- Goals:
  - デザイナーが 1 つのワールドシーンで作業し、ビルド時に自動でセル分割される「オーサリングと実行の分離」
  - フレーム予算内での非同期ストリーミング(ヒッチなし: 1 フレームあたりのメインスレッド消費を制御)
  - HLOD による遠景の描画コスト削減とポップイン抑制
  - Data Layers によるクエスト/イベント単位のコンテンツ切替
  - すべての機能がエディタツール + ランタイム API + デバッグ表示の 3 点セットを持つ
- Non-Goals:
  - 地形生成・植生生成(既存の Unity Terrain / サードパーティに委譲)
  - セーブ/ロードの永続化システム(別プロポーザル)
  - ネットワーク同期の実装(抽象化のみ行う)
  - ECS/DOTS ベースの実装(GameObject ベースで開始、ホットパスのみ Jobs/Burst)

## Decisions

### D1: セル分割方式 — 2D 均一グリッド + 単一レイヤー

- ワールドを XZ 平面の均一グリッド(既定 256m、設定可能)で分割する。UE の複数グリッドレイヤーは初期実装では単一グリッドに簡略化し、大型オブジェクトは「Always Loaded セル」に配置する
- 代替案: 四分木(実装コスト高・デバッグ困難)、Sector ベース手動分割(オーサリングコスト高)→ 均一グリッドが最も予測可能でツール化しやすい

### D2: オーサリングと実行の分離 — ベイクパイプライン

- デザイナーはワールドシーン(オーサリングシーン)で自由に配置する。エディタのベイク処理が静的オブジェクトをセル境界で分類し、セルごとの Addressables プレハブ(セルコンテンツ)を生成する
- 動的/常駐オブジェクト(プレイヤー、マネージャ類)は Persistent シーンに残す
- 代替案: セルごとの手動サブシーン管理(Multi-Scene 編集)→ 人的ミスが多く却下。ECS SubScenes → Non-Goal

### D3: ロード単位 — Addressables プレハブ(シーンではなく)

- セルは Addressable なプレハブとして `InstantiateAsync` でロードする。シーン加算ロードよりハンドル管理・依存分析・メモリ解放(`Release` で即時)の制御性が高い
- 静的バッチング相当は StaticBatchingUtility をセルインスタンス化直後に適用
- 代替案: `SceneManager.LoadSceneAsync` 加算 → ライティングシーン設定の分離が煩雑、アンロードが重い

### D4: ストリーミング判定 — StreamingSource 抽象 + 距離リング

- `IStreamingSource`(位置・半径・優先度を持つ)を登録制にする。既定はプレイヤー 1 つだが複数登録可能(将来の MP・カットシーンカメラ・高速移動の先読みに対応)
- セル状態は `Unloaded → Loading → Loaded → Activated`(UE と同じ 2 段階)。ロード半径 > アクティベート半径 とし、境界のヒステリシス(既定 32m)で振動を防ぐ
- 判定はセル中心距離ではなく AABB 最近点距離を用いる

### D5: 非同期処理とフレーム予算

- ロード/アンロード要求は優先度付きキュー(距離昇順、ソース優先度加味)で処理。同時 in-flight ロード数(既定 4)と 1 フレームあたりのインスタンス化予算(既定 2ms、`Time.realtimeSinceStartup` 計測)を制御
- Unity 6 の `Application.backgroundLoadingPriority` と Incremental GC を前提設定に含める

### D6: HLOD — セル単位のオフラインベイク

- ベイク時にセルごとの静的メッシュを結合 + テクスチャアトラス化した HLOD メッシュを生成(Level0: メッシュ結合、Level1: ビルボードインポスタは将来拡張)
- セルが `Loaded` 未満のとき HLOD を表示し、`Activated` への遷移完了フレームでスワップ(1 フレームの重複表示を許容しポップを隠す)
- HLOD 自体も Addressable とし、HLOD 表示半径(既定: ロード半径の 4 倍)を持つ
- 代替案: サードパーティ(AutoLOD 等)→ 依存を避け、Mesh API + `Mesh.CombineMeshes` ベースで自前実装

### D7: Data Layers — アセット定義 + ランタイムマスク

- `DataLayerAsset`(ScriptableObject)でレイヤーを定義し、オーサリング時にオブジェクトへ `DataLayerTag` コンポーネントで付与。ベイク時にセル内でレイヤー別サブツリーに分類される
- ランタイムはレイヤーの `Enabled/Disabled` を切替可能。セルロード時は有効レイヤーのみインスタンス化(または SetActive)する
- 代替案: Unity の Layer/Tag 流用 → 数の上限と意味の混同があり却下

### D8: アセンブリ構成

- `OpenWorld.Runtime`(ランタイム全機能)、`OpenWorld.Editor`(ベイク・ツール)、`OpenWorld.Tests`(EditMode/PlayMode)の 3 asmdef。Editor から Runtime へ一方向参照

## Risks / Trade-offs

- ベイク時間の増大(大規模ワールド)→ 差分ベイク(セル単位のダーティフラグ)を仕様に含める
- Addressables の依存重複によるメモリ肥大 → ベイク後に Analyze ルールで重複検出し共有バンドル化。デバッグ HUD にメモリ統計を表示
- インスタンス化スパイク → フレーム予算 + プレハブの階層深さ制限(ベイク時警告)で緩和
- OneDrive 同期下のプロジェクト → Library/ベイク成果物は同期対象外を推奨(運用注意として明記)
- 単一グリッドの制約 → 巨大オブジェクト(山・大型建造物)は Always Loaded セルへ。将来複数グリッドレイヤーを ADDED で拡張可能な設計

## Migration Plan

新規追加のため移行はなし。既存シーンをオープンワールド化する場合は、オーサリングシーンへオブジェクトを移し、ベイクを実行するのみ。ロールバックはベイク成果物(`Assets/OpenWorld/Generated/`)の削除で完結し、オーサリングシーンは常に無傷。

## Open Questions

- HLOD のテクスチャアトラス解像度の既定値(2048 想定、実測で決定)
- ストリーミング判定の Jobs 化の閾値(セル数 > 1000 で必要か、プロファイル後に判断)
- Terrain のセル分割対応(Unity Terrain のタイル分割を本仕様に含めるか、別プロポーザルか)
